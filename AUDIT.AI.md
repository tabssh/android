## Pass 8: Terminal Subsystem Deep-Dive
Targeted audit of `terminal/` directory plus `ui/views/TerminalView.kt` and `terminal/TermuxBridge.kt`. Categories covered: touch/gesture handling, drawing/rendering, accessibility, keyboard/IME, lifecycle, threading, memory, duplicate-vs-shared-utils. Findings below — all fixed in this commit.

- `ui/views/TerminalView.kt` accessibility node — `onInitializeAccessibilityNodeInfo` only populated `info.text` from the legacy `terminalBuffer`, which is null whenever a Termux/SSH session owns the view. Result: TalkBack saw an empty terminal node for every real SSH tab. Now prefers `termuxBridge?.getScreenContent()` (which uses Termux's `screen.getSelectedText` honouring line-wrap flags), and falls back to the legacy buffer for the in-memory emulator path.
- `ui/views/TerminalView.kt` dead fields — `accessibilityHelper`, `lastRenderedRows`, `lastRenderedCols`, `lastCursorRow`, `lastCursorCol`, `rowTextBuilder`, `clipRect` were all declared but never read or written after initial assignment. Removed; the `dirtyRows` BitSet is kept (with a clarifying comment) because rows-dirty marking is still useful infrastructure even though `invalidate(left,top,right,bottom)` is deprecated on modern Android.
- `ui/views/TerminalView.kt` `TerminalAccessibilityHelper` class — fully orphan helper class at end of file. Field declaration referencing it was already removed; the class itself was never instantiated and was dragging in an unused `AccessibilityAction` data class. Deleted entirely.
- `terminal/emulator/TerminalEmulator.kt` dead code — `processChar()`, `writeChar()`, `processControlChar()`, `handleKeyPress()`, `handleControlKey()`, `handleAltKey()`, `handleNormalKey()`, and `sendKeyPress()` were leftovers from the pre-ANSIParser era. All terminal input is now routed through `ansiParser.processInput()` (which writes to the buffer directly), and all key sequencing happens in `TermuxBridge.sendKeyPress()`. Cross-referenced every caller via grep — `SSHTab.sendKeyPress()` calls the **bridge** method, not the emulator's. Removed the dead methods plus the now-unused `cursorX`/`cursorY` mirror state that nothing ever read externally.
- `ssh/connection/SSHConnection.kt` `_bytesTransferred` wired — `getInputStream()` / `getOutputStream()` now return cached counting wrappers that increment `_bytesTransferred` on every successful `read()` / `write()`. The wrappers are recreated whenever the underlying JSch stream identity changes (new shell/exec channel after reconnect) and cleared in `disconnect()` so no stale raw-stream pointers survive a teardown. `ConnectionStats.bytesTransferred` and any `bytesTransferred` StateFlow observer now sees real traffic counts instead of a constant 0. Confirmed the wrap point is sufficient: all shell I/O in the app (Termux bridge, SSHTab read loop, exec channel consumers) routes through these two accessors — no direct `shellChannel.inputStream` access outside `openShellChannel()` (where the pre-`connect()` stream allocation is intentional and not part of the byte-count path).

## Pass 9: SSH Subsystem Deep-Dive
Targeted audit of `ssh/` directory (SSHConnection.kt 2167 lines, SSHSessionManager.kt 440 lines, auth/, forwarding/, connection/). Categories: auth flow completeness, error propagation, stale session reuse, port/X11 forwarding teardown, reconnect/retry, thread safety, resource leaks, byte-counter wiring, duplicate identity resolution.

Outcome: the SSH subsystem is in good shape from prior passes — stale pool entries are discarded via the `stateOk && sessionOk` gate in `SSHSessionManager.createConnection()`, `disconnect()` tears down `connectJob` / reconnector / `openChannels` / `sftpChannel` / `x11Proxy` / `session` / `jumpHostSession` in the correct order, port forwarding `cleanup()` issues per-tunnel JSch `unforward` calls before cancelling the scope, X11 proxy handles leaked-fd protection on connect failures, and `executeCommand()` has a try/finally channel-leak guard. Only the `_bytesTransferred` wiring required code change (now fixed — see above).

Duplicate identity resolution between `TabTerminalActivity.startConnection()` (line 1666 — resolves identity for password-prompt gating BEFORE connect) and `SSHConnection.connect()` (line 239 — resolves it again for its own credential pipeline) is intentional: the TabTerminalActivity needs `effectiveAuthType`/`effectiveKeyId` to decide whether to show a passphrase dialog before the connect attempt, and `SSHConnection.connect()` is invoked from many other code paths (monitoring, background, host availability) that have no UI to consult. Consolidating would force every non-UI caller to pre-resolve an `Identity`, increasing surface area for stale-cache bugs. Leaving as-is.

- `terminal/TermuxBridge.kt` OSC 8 link cap — `while (osc8Links.size > 200) osc8Links.removeAt(0)` on a `CopyOnWriteArrayList` is O(N) per element removed (each removal allocates a fresh backing array), so a flood that produced 200+ links in one append turned into O(N^2). Replaced with a single-allocation rotation: when the cap is reached, build a fresh list containing the last (CAP-1) entries plus the new one and swap the volatile reference in one atomic write. Readers on the UI thread still see a consistent snapshot. Extracted `OSC8_LINK_CAP = 200` to the companion object.

## Pass 11: Cloud + Cluster Deep-Dive
Targeted audit of `app/src/main/java/io/github/tabssh/cloud/` (10 files, 2040 lines) and `app/src/main/java/io/github/tabssh/cluster/` (1 file, 176 lines). Categories: API error handling, state-machine correctness, credential storage, polling lifecycle, UI wiring, empty states, duplicate code, thread safety, resource leaks. All findings fixed in this commit.

- `cloud/GcpComputeClient.kt` + `cloud/AzureVmClient.kt` `cachedInstances` — power actions (start/stop/restart) need the per-instance zone (GCP) / resourceGroup (Azure) which lives in `cachedInstances` populated by `fetchLiveInstances`. `CloudAccountManagerActivity` previously created a fresh `CloudProvider` instance per action via `providerType.newClient()`, so the second client always had an empty `cachedInstances` and every GCP/Azure power action silently returned false. Fixed two ways: (1) cache one provider client on the Activity keyed by provider tag and reuse it across loadInstances + all power/restart calls; (2) marked the `cachedInstances` field `@Volatile` in both clients so the read in the action method on a worker thread always sees the latest write from the list fetch on a different worker.
- `cloud/CloudProvider.kt` typed auth errors — added `CloudAuthException : RuntimeException`. Every cloud client (DigitalOcean, Hetzner, Linode, Vultr, AWS, GCP, Azure) now throws it on HTTP 401/403 from both list and action endpoints. `CloudAccountManagerActivity.loadInstances`/`performPowerAction`/`performRestartAction` and `CloudAccountsFragment.refreshAccount` catch it first and show "Token invalid — re-add account" instead of the generic "Load failed: HTTP 403" — actionable message, user knows the credential is bad rather than the network. AzureVmClient's `azureVmAction` had a `catch (_: Exception)` that swallowed every error from `exchangeForAccessToken`; now re-throws CloudAuthException so the UI can react.
- `cluster/ClusterCommandExecutor.kt` `cancelAll()` no-op — `executionJob` was declared but never assigned, so the UI's Cancel button (`ClusterCommandActivity.cancelExecution`) called `executor.cancelAll()` which logged "All executions cancelled" without cancelling anything. The fan-out kept running and the user was stuck waiting through the full timeout. Now publishes `coroutineContext[Job]` inside the `coroutineScope` so `cancelAll()` actually cancels the parent and all `async` children; cleared in `finally` so a stale reference doesn't survive a clean run. Field is `@Volatile` because the assignment runs on an IO worker and the cancel comes from the UI thread.
- `cloud/AwsEc2Client.kt` region validation — `fetchInventory` checks `if (region.isBlank()) throw …` but `fetchLiveInstances` and `awsInstanceAction` previously trusted the token format and would build `https://ec2..amazonaws.com/` URLs with an empty region (DNS resolution failure or worse — wildcard match against a typo'd region). Added the same region-blank guard to both other entry points so all three paths fail loudly with the same error.
- `ui/fragments/CloudAccountsFragment.kt` + `res/layout/item_cloud_account.xml` dead `switch_enabled` — the layout has a `SwitchCompat` per row and the `CloudAccount` entity has an `enabled: Boolean = true` column (v23+ migration), but the bind code did `isChecked = true` + null listener so the user could never actually disable an account. Wired it: listener writes the new state via `CloudAccountDao.update(account.copy(enabled = …))` from the viewLifecycleOwner scope, `refreshAccount` early-returns with a "${name} is disabled — enable to refresh" toast for disabled rows. Tap-to-open the instance manager intentionally still works for disabled accounts — the toggle suppresses automated polling, never blocks explicit navigation. Listener is detached BEFORE updating `isChecked` so DiffUtil-driven rebinds don't fire spurious toggle callbacks.
- `ui/activities/CloudAccountsActivity.kt` switch_enabled — this `@Deprecated` activity is no longer reachable from the live UI (Pass 5 already flagged it) but still defines the same SwitchCompat in its bind code. Set the read-only `account.enabled` value and disabled the switch with `isEnabled = false` so a screenshot of the deprecated activity doesn't mislead a developer into thinking it's an editable control.
- `cloud/DigitalOceanClient.kt` / `HetznerClient.kt` / `LinodeClient.kt` / `VultrClient.kt` action endpoints — the post*Action helpers previously returned `resp.code == 2xx || resp.isSuccessful` without distinguishing 401/403 from a generic server error, so an expired token gave a confusing "Failed to start instance" with no hint to re-add. Now `CloudAuthException` is raised on 401/403 first, then the success-code check runs.

## Pass 12: Hypervisor Subsystem Deep-Dive
Targeted audit of `app/src/main/java/io/github/tabssh/hypervisor/` (Proxmox, XCP-ng + Xen Orchestra, VMware, libvirt, OCI, VNC console, rfb/). Categories covered: API client completeness (no stub power/list/IP ops), RFB state machine and pseudo-encoding handling, WebSocket VNC binary frame reassembly, SSH tunnel teardown for libvirt VNC, credential storage (HypervisorPasswordStore / OCI key material / VeNCrypt), error propagation (XML-RPC faults, REST 401/403/5xx, virsh stderr, OCI signature errors), polling cancellability, RFB IO/Main thread safety, resource leak audit on every error path (socket / WebSocket / SSH channel / PixelFormat buffers), VM list UI empty state and per-row state dots. Files read in full: `ProxmoxApiClient.kt`, `LibvirtApiClient.kt`, `LibvirtVm.kt`, `XCPngApiClient.kt`, `XenOrchestraApiClient.kt` (all 1458 lines), `VMwareApiClient.kt`, `OciApiClient.kt`, `OciSigner.kt`, `OciKeyMaterial.kt`, `OciConfigParser.kt`, `OciInstance.kt`, `VncDirectConnector.kt`, `VncStreamHolder.kt`, `VncConsoleChannel.kt`, `PixelFormat.kt`, `RfbConstants.kt`.

- `hypervisor/xcpng/XCPngApiClient.kt` XML-RPC value escaping — `authenticate()` and `buildXmlRequest()` were string-interpolating `$username` / `$password` / `$sessionId` / each `$param` directly into the XML-RPC body with no escaping. Any value containing `&`, `<`, `>`, `'`, or `"` produced malformed XML and the XAPI server returned a parse fault with no useful context — visible failures included a password containing `&` rejected as "Auth fail" even when correct, and any VM name with `<` / `>` round-tripped through `parseStringResponse` corrupted. Worse, an attacker-controlled password of the form `</value><value>OpaqueRef:…` would have been parsed by the server as two separate string params instead of one, allowing forged-param injection into login_with_password. Added a `xmlEscape()` helper applying the five XML 1.0 entity references and wrapped every interpolated value: username, password, method name, sessionId, and each request param. Added a paired `xmlUnescape()` and applied it to `parseStringResponse()` so XAPI responses carrying the same entities round-trip correctly to the UI (e.g. a VM named "A & B" no longer surfaces as "A &amp; B" in the manager list). `parseArrayResponse()` returns OpaqueRef strings that contain only `[A-Za-z0-9:-]` so are left as-is — they are immediately re-escaped by `buildXmlRequest()` on the next call.

## Pass 13: SFTP + Backup + Sync Deep-Dive
Targeted audit of `app/src/main/java/io/github/tabssh/sftp/`, `.../backup/`, and `.../sync/`. Categories: SFTP transfer error/progress/cancel paths, editor save race, path traversal, empty state UI, chmod; backup export coverage and AES-256 encryption, import schema/duplicate/corrupt-ZIP handling, validator wiring; 3-way merge correctness, sync encryption + PBKDF2, observer lifecycle, worker constraints. Files read in full: `sftp/SFTPManager.kt`, `sftp/TransferTask.kt`, `sftp/SCPClient.kt`, `backup/BackupManager.kt`, `backup/export/BackupExporter.kt`, `backup/import/BackupImporter.kt`, `backup/validation/BackupValidator.kt`, `sync/SAFSyncManager.kt`, `sync/encryption/SyncEncryptor.kt`, `sync/data/SyncDataCollector.kt`, `sync/data/SyncDataApplier.kt`, `sync/merge/MergeEngine.kt`, `sync/merge/ConflictResolver.kt`, `sync/metadata/SyncMetadataManager.kt`, `sync/models/SyncModels.kt`, `sync/observer/DatabaseChangeObserver.kt`, `sync/worker/SyncWorker.kt`.

- `sftp/SFTPManager.kt` resume-download truncation — the resume branch of `downloadFile()` called `localFile.outputStream().close()` (which truncates the partial file to zero bytes) and then routed writes through an `appendOutputStream()` extension that was a stub aliasing back to `outputStream()` — so it *also* truncated. Net effect: every "resume from offset N" actually discarded the on-disk prefix and wrote only the tail of the stream, producing a corrupt file the user thought had been recovered. Replaced both calls with explicit `FileOutputStream(localFile, append=startOffset>0)` so the post-skip bytes append onto the already-downloaded prefix as JSch's `ChannelSftp.RESUME` requires. Added `import java.io.FileOutputStream` and removed the stub extension function so no future caller can re-introduce the same data-loss bug. Fresh downloads still pass `append=false`, matching the prior intent.
- `backup/validation/BackupValidator.kt` sub-result `isValid` hardcoded false — `validateConnectionsData`, `validateKeysData`, `validatePreferencesData`, and `validateThemesData` each ended with `ValidationResult(false, errors, warnings)`, so any caller that consumed the per-section `isValid` flag (instead of inspecting `errors`) treated a clean backup as invalid. The outer aggregator happened to use `errors.isEmpty()` so the user-visible flow worked, but the inconsistency would silently break any future caller. Changed all four to `ValidationResult(errors.isEmpty(), errors, warnings)`.
- `backup/validation/BackupValidator.kt` `isBackupEncrypted` stale heuristic — checked "JSON fails to parse AND matches a base64 regex". `SyncEncryptor` (the v3 backup cipher) emits a raw binary stream that begins with the literal magic header `TABSSH_SYNC_V2` followed by salt + nonce + AES-GCM ciphertext — never base64 — so every encrypted backup was misreported as plaintext. UI callers downstream skipped the passphrase prompt and tried to parse the binary as JSON, throwing a confusing parser error. Rewrote with a 14-byte magic-header check against `TABSSH_SYNC_V2` (`Charsets.ISO_8859_1` so the byte-for-byte compare is correct regardless of platform default charset) and added a `String` overload that re-encodes via ISO-8859-1 for legacy callers still holding a String view of the binary blob. Matches the exact format produced by `SyncEncryptor.encrypt()`.

Observed but intentionally NOT changed:
- `sync/observer/DatabaseChangeObserver.kt` watches a subset of tables (connections, ssh_keys, themes, snippets, command_history, security_settings). The debounce-driven `SyncDataCollector` reads ALL ~17 entity types when triggered, so a change in any of the watched tables is sufficient to capture changes anywhere else that happens in the same transactional window. Adding the remaining tables to the observer would only multiply trigger frequency without changing the snapshot content.
- `sync/worker/SyncWorker.kt` is upload-only. Two-way sync is delivered via the explicit `SAFSyncManager.sync()` path which runs the 3-way merge; the background worker exists to keep the user's SAF location current with their device after offline edits. This is by design and documented in `IDEA.md §7`.

## Pass — 2026-06-15 — Targeted deep-dive: `crypto/` + `storage/`

Categories audited: (1) key generation security, (2) AES-GCM IV/tag/AAD, (3) Android Keystore (hardware-backed, StrongBox, user-auth), (4) FIDO completeness, (5) TLS hostname verification, (6) key import parsers, (7) passphrase memory zeroing, (8) constant-time comparison, (9) Room migration chain integrity, (10) DAO correctness, (11) entity columns vs migrations, (12) preference key/XML coverage, (13) file management cleanup/locking.

Fixed:
- `storage/database/TabSSHDatabase.kt` — added `fallbackToDestructiveMigrationFrom(1, 2)`. Without it, any leftover alpha v1/v2 install would crash at first open on upgrade to v3 because the addMigrations() list is empty and Room defaults to throw. Only v1 and v2 are listed (no real user data ever shipped) — from v3 forward, every bump MUST register a real Migration. Updated kdoc to spell this out so a future contributor doesn't append v3, v4, … to the destructive list. `app/schemas/` confirms only `3.json` exists.
- `crypto/keys/KeyType.kt` — removed `1024` from DSA `supportedKeySizes`. 1024-bit DSA is ~80-bit security, rejected by current OpenSSH and disallowed by NIST since 2013. UI dropdowns would have offered users a value that ssh-keygen and every modern server would refuse. Now `listOf(2048, 3072)` with an inline kdoc explaining why.
- `crypto/SSHKeyGenerator.kt` — the encrypted OpenSSH-v1 export path was format-invalid: it claimed `aes256-cbc` / `kdf=bcrypt` in the header, then encrypted the private section with raw `AES/CBC/PKCS5Padding` and a key derived by `passphrase.toByteArray().copyOf(32)` (zero-pad/truncate, no KDF). The result would be rejected by every OpenSSH client. Rerouted all passphrase-protected exports through the existing `exportPrivateKeyPEM` path (BouncyCastle's `JcePEMEncryptorBuilder("AES-256-CBC")` → PKCS#8 PEM). OpenSSH reads PKCS#8 PEM natively (it is the historical default before `-O openssh-key-v1`), so interoperability is preserved. Deleted the broken `encryptOpenSSHPrivateKey` helper and the unused `Cipher`/`IvParameterSpec`/`SecretKeySpec` imports. The unencrypted v1 branch keeps writing `cipher = none`, `kdf = none`, empty kdf-options.
- `crypto/SSHKeyParser.kt` — `bcryptPbkdf` was labelled "OpenSSH KDF" but actually called `PBKDF2WithHmacSHA256` and logged a "fallback" warning. PBKDF2 produces fundamentally different key material than OpenSSH's bcrypt_pbkdf — decryption would silently produce garbage and surface as a misleading "wrong passphrase" error. A correct bcrypt_pbkdf needs the OpenBSD bcrypt-hash core (32-byte output, little-endian, "OxychromaticBlowfishSwatDynamite" magic), which BouncyCastle 1.79 does not expose. Hand-rolling Blowfish + EksBlowfishSetup for an import path is a crypto liability we won't ship. Replaced with `UnsupportedOperationException` carrying an actionable message — convert the key via `ssh-keygen -p -m PEM -N "" -f keyfile` then re-import. This is honest about the gap and matches the export-side decision.
- `crypto/SSHKeyParser.kt` — PuTTY v2 decryption used MD5 with ASCII counter characters (`"0" + passphrase`). The actual PuTTY v2 KDF is SHA-1 with a 4-byte BIG-ENDIAN counter prefix: `key = SHA-1(0x00000000 || pass) || SHA-1(0x00000001 || pass)`. MD5 + ASCII counter would never have decrypted a real .ppk. Rewritten to SHA-1 + binary counter, truncating the 40-byte SHA-1||SHA-1 output to 32 for AES-256. Added full Private-MAC verification: HMAC-SHA-1 over `lenpref(keytype) || lenpref(encryption) || lenpref(comment) || lenpref(public) || lenpref(private)` with `macKey = SHA-1("putty-private-key-file-mac-key" || passphrase)`, compared constant-time. Without MAC verification a tampered private blob could pass parsing and emit a corrupted key. PuTTY v3 unencrypted handled with the spec's empty-string-SHA-256 MAC key; v3 encrypted (Argon2) rejected with a clear error since we do not ship Argon2.
- `crypto/keys/KeyStorage.kt` — `parsePuttyKey` was a stub returning `ParseResult.Error("PuTTY key format not yet supported…")` while the `KeyFormat.PUTTY_PRIVATE` enum value was already wired into format detection. Users who imported a .ppk hit a dead end. Replaced with delegation to the now-correct `SSHKeyParser.parse`. Returns proper KeyPair + comment, or maps the parser's IllegalArgumentException into ParseResult.Error.
- `crypto/keys/KeyStorage.kt` — `extractCommentFromPEM` looked for `#`-prefixed lines. PEM (RFC 7468) has no comment convention; ssh-keygen, openssl, and BouncyCastle never emit `#` lines. The function always returned empty for real PEM but the code path obscured that. Replaced with a one-liner returning `""` plus a kdoc explaining why and pointing callers to use the file path or user input for the comment.
- `crypto/storage/SecurePasswordManager.kt` — four policy fields (`passwordTTL`, `requireBiometricForSensitive`, `maxFailedAttempts`, `autoDeleteOnFailure`) were written by `setSecurityPolicy()` but never read anywhere in the class. The accessor methods that would have consumed them never existed; persisted policy lives in `PreferenceManager`. Kept the field for `defaultStorageLevel` (which IS read in `savePassword` fallback) and dropped the four dead fields. Kept `setSecurityPolicy` signature for ABI/test compatibility (the unit test passes all five args) and added `@Suppress("UNUSED_PARAMETER")` plus a kdoc that explicitly says only `defaultLevel` is honoured and the rest live in PreferenceManager. Logging line updated to match.
- `storage/files/FileManager.kt` — deleted. Zero callers across `app/src` (verified with grep). `setReadable(false, false)` / `setWritable(false, false)` on a `java.io.File` are no-ops under Android's app-private storage anyway (Android enforces permissions at the kernel level via UID, not POSIX bits), and the only "lock file" mechanism was a sibling `.lock` file that nothing else consulted. Dead code on a security-sensitive path is worse than absent code. Removed file and now-empty parent directory `storage/files/`.

Reviewed and intentionally NOT changed:
- `crypto/tls/HypervisorTrustManagerFactory.kt` (243 lines) — TOFU pinning with system-CA fallback for hypervisor REST/WebSocket clients. Hostname verification is intentionally bypassed (kdoc documents why: hypervisor TLS certs routinely self-sign with a CN/SAN that doesn't match the user's bookmarked hostname; pinning replaces hostname checks).
